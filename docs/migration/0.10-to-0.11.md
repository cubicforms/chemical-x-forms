# Migrating from 0.10.x to 0.11.x

Three breaking changes, all in the same spirit: Vue auto-unwraps
refs only when they're top-level on a component's setup return.
Refs nested inside an API _object_ — like `useForm()`'s return — do
NOT unwrap, and the previous API made consumers pay for that
distinction. 0.11 pushes the unwrap into the library.

Two changes you'll actually feel; one rename that only shows up if
you were importing an internal type.

## Breaking: drop `.value` from `fieldErrors`

**0.10:** `fieldErrors: Readonly<ComputedRef<FormFieldErrors<Form>>>`
— consumers had to write `fieldErrors.value.email` in script and
`fieldErrors.email` in templates (the latter worked because the
compiler auto-unwrapped the outer ref). Asymmetric and surprising.

**0.11:** `fieldErrors: Readonly<FormFieldErrors<Form>>` — a Proxy
view over the same underlying `ComputedRef`. Reads from script and
templates go through the same surface; both dot-access directly.

```diff
- const firstEmailError = fieldErrors.value.email?.[0]
+ const firstEmailError = fieldErrors.email?.[0]
```

Watchers that used to subscribe to the ref object itself must
switch to the getter form (a `Ref` is no longer on the surface):

```diff
- watch(api.fieldErrors, (next) => /* … */)
+ watch(() => api.fieldErrors.email, (next) => /* … */)
```

Read-only enforcement is preserved at compile time (the type) and
at runtime (the Proxy rejects writes with a dev-mode console warn
pointing at `setFieldErrors` / `clearFieldErrors`).

## Breaking: 9 form-level scalars moved to `state`

The previous release returned `isDirty`, `isValid`, `isSubmitting`,
`isValidating`, `submitCount`, `submitError`, `canUndo`, `canRedo`,
and `historySize` as top-level `Readonly<ComputedRef<X>>` fields.
Templates that bound to them directly — `:disabled="isSubmitting"`
— bound to the ComputedRef _object_ (always truthy), not its
`.value`. The form was permanently disabled and nothing in the type
system flagged it.

**0.11:** the 9 scalars are bundled on a single `state` property
backed by `reactive()` + `readonly()`. Vue's reactive object
auto-unwraps nested refs at property access, so the templates work
by default and scripts drop `.value`.

```diff
  // Script:
- if (form.isDirty.value && form.isValid.value) { /* … */ }
+ if (form.meta.isDirty && form.meta.isValid) { /* … */ }

  // Template:
- <button :disabled="isSubmitting">
+ <button :disabled="state.isSubmitting">
```

Destructure `state` at the top of your setup if you're accessing
several leaves:

```vue
<script setup>
  const { register, handleSubmit, fieldErrors, state } = useForm({ schema })
</script>
```

`undo()` and `redo()` remain at the top level — they're methods,
not state.

### Migration one-liner

For most consumers, a single `sed` pass handles the rewrite:

```bash
# From the root of a attaform-style project:
grep -rlE '\.(isDirty|isValid|isSubmitting|isValidating|submitCount|submitError|canUndo|canRedo|historySize)\.value' \
  --include='*.ts' --include='*.vue' . \
  | xargs sed -i '' -E \
    's/\.(isDirty|isValid|isSubmitting|isValidating|submitCount|submitError|canUndo|canRedo|historySize)\.value/.state.\1/g'
```

(Drop the `''` after `-i` on GNU sed / Linux.)

**Vue templates need a parallel edit** for the `.value`-less
pattern: `{{ isSubmitting }}` → `{{ state.isSubmitting }}`. Grep
for bare references and adjust by hand — the sed rewrite above
covers `.X.value` but not the unqualified template bindings.

Watchers on the old refs:

```diff
- watch(form.isSubmitting, (submitting) => /* … */)
+ watch(() => form.meta.isSubmitting, (submitting) => /* … */)
```

The getter-form is also the recommended pattern for any future-
proof `state.*` watch.

### The new `FormMeta` type

`useForm().state` is typed as the exported `FormMeta` interface:

```ts
import type { FormMeta } from 'attaform'

// or via the generic return type:
type State = ReturnType<typeof useForm<Schema>>['state']
```

Fields are `readonly` at the type level; reactive() + readonly()
reject writes at runtime with a dev-mode warning.

## Breaking (rare): internal `FormMeta` renamed to `FormStore`

If you imported `FormMeta` from `attaform/runtime/core/
create-form-state` — a deep path we never advertised as public —
it's now `FormStore` from `./create-form-store`. The module file
was renamed too.

No public entry re-exported the old internal `FormMeta`, so this
only catches consumers who deliberately reached past the public
surface. The new public `FormMeta` interface (above) is unrelated.

## Breaking: `initialState` renamed to `defaultValues`

The config key that seeds the form with starting field values is
now `defaultValues` — matching React Hook Form's vocabulary and,
more importantly, removing the naming collision with the new
`form.meta` property. Reading
`useForm({ schema, initialState: {...} })` left a reasonable doubt
about whether the value seeded `form.meta` (the new flag bundle)
or the schema's field values; `defaultValues` is unambiguous.

```diff
  useForm({
    schema,
-   initialState: { email: 'alice@example.com' },
+   defaultValues: { email: 'alice@example.com' },
  })
```

`reset()`'s positional parameter was renamed for the same reason
(`nextInitialState` → `nextDefaultValues`). Positional callers are
unaffected; only doc-readers / destructurers notice.

### Migration one-liner

```bash
grep -rl 'initialState\|InitialState\|initial-state' \
  --include='*.ts' --include='*.vue' --include='*.md' . \
  | xargs sed -i '' \
    -e 's/InitialState/DefaultValues/g' \
    -e 's/initialState/defaultValues/g' \
    -e 's/initial-state/default-values/g'
```

(Drop the `''` after `-i` on GNU sed / Linux.)

### Custom `AbstractSchema` adapter authors

The matching method on the adapter contract was renamed alongside
the config key:

| Before                        | After                          |
| ----------------------------- | ------------------------------ |
| `getInitialState(config)`     | `getDefaultValues(config)`     |
| `InitialStateResponse<Form>`  | `DefaultValuesResponse<Form>`  |
| `GetInitialStateConfig<Form>` | `GetDefaultValuesConfig<Form>` |

The deep-path module `runtime/adapters/zod-v4/initial-state` was
renamed to `runtime/adapters/zod-v4/default-values`. Only relevant
if you imported the internal helper `getDefaultValuesFromZodSchema`
(formerly `getInitialStateFromZodSchema`); the public adapter
surface re-exports it through `attaform/zod`, which is
unchanged at the entry-point level.

Custom adapters built against `AbstractSchema` need a one-method
rename:

```diff
  const myAdapter: AbstractSchema<Form, Form> = {
    fingerprint() { /* … */ },
-   getInitialState(config) { /* … */ },
+   getDefaultValues(config) { /* … */ },
    getSchemasAtPath(path) { /* … */ },
    validateAtPath(data, path) { /* … */ },
  }
```

The same sed one-liner above handles this, since both the config
key and the method name share the `InitialState` / `initialState`
substring.

## What's unchanged

- Every other part of the API. `register`, `handleSubmit`,
  `getValue`, `setValue`, `reset`, `resetField`, `getFieldState`,
  `validate`, `validateAsync`, the typed array helpers, and the
  opt-in options (`onInvalidSubmit` / `fieldValidation` / `persist`
  / `history`) all keep their 0.10 signatures.
- Watchability. `watch(() => form.meta.isSubmitting, …)` tracks
  the underlying ComputedRef through the reactive wrapper exactly
  the way `watch(form.isSubmitting, …)` did in 0.10.
- Read-only semantics. `state.*` writes are rejected at runtime;
  the TypeScript interface enforces `readonly`.
