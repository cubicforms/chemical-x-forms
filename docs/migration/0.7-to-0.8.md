# Migrating from 0.7.x to 0.8.x

Four changes consumers touch, and one big set of additions. Most
0.7 code compiles unchanged; the three changes that DO affect
compilation are covered first.

## Breaking: `key` is now required on `useForm`

**0.7:** `key` was recommended. Consumers who forgot it got "shared
state" quirks where two forms ended up pointing at the same
registry slot.

**0.8:** `key` is required at the type level. Forgetting it is a
compile error:

```ts
// Compile error under 0.8:
useForm({ schema })
// Fix — give it a key unique within the app:
useForm({ schema, key: 'signup' })
```

The runtime still throws a clear error if a non-TS caller passes
`undefined`, `null`, or `''` (see `test/composables/use-abstract-form.test.ts`),
so JS consumers aren't left guessing either.

## Breaking: `@chemical-x/forms/zod-v3` vs `@chemical-x/forms/zod`

The Zod adapter is now split across two subpaths:

- `@chemical-x/forms/zod` — zod v4 (recommended for new projects)
- `@chemical-x/forms/zod-v3` — zod v3 (legacy)

If you were importing from `@chemical-x/forms/zod-v3` already, nothing
changes. If you were importing from a bare `@chemical-x/forms` entry
and relying on a built-in zod adapter, pick the subpath matching your
installed zod version.

The two adapters are physically isolated — the v3 path does not
import zod v4 internals and vice versa. Uninstalling the zod major
you don't use is safe.

## Breaking: `AbstractSchema.validateAtPath` is now async

**0.7:** `validateAtPath(data, path)` returned a synchronous
`ValidationResponse`. Async zod refinements (`z.refine(async ...)`)
were not supported.

**0.8:** `validateAtPath` returns `Promise<ValidationResponse>`.
Both zod adapters use `safeParseAsync` under the hood, so async
refinements are a first-class schema feature.

Consumers who were only using `handleSubmit` / `useForm` need no code
changes — those APIs `await` internally. But:

- **Custom `AbstractSchema` implementations** (anyone who wrote
  their own adapter for Valibot / ArkType / custom rules) need to
  make `validateAtPath` return a Promise. Wrap sync bodies with
  `async` / `Promise.resolve(...)`; the core doesn't care whether the
  parse is sync-under-the-hood or truly async.

- **`validate()`'s ref shape changed.** It now carries a `pending`
  flag so templates can distinguish "in-flight" from "settled":
  ```ts
  const status = validate()
  // Previously:
  if (status.value.success) { /* ... */ }
  // Now — gate on pending first:
  if (!status.value.pending && status.value.success) { /* ... */ }
  ```
  Stale in-flight runs are dropped internally via a generation
  counter — only the latest run writes to the ref.

- **New API: `validateAsync(path?)`** for imperative one-shot calls,
  returning `Promise<ValidationResponseWithoutValue<Form>>`.

- **New API: `isValidating`** — reactive boolean, `true` while any
  validation call is in flight.

- **New API: `PendingValidationStatus` / `SettledValidationStatus`
  / `ReactiveValidationStatus<Form>`** — the three type names the
  reactive `validate()` ref's value conforms to. `PendingValidationStatus`
  is the `pending: true` variant; the settled shape is the old
  `ValidationResponseWithoutValue` + a `pending: false` discriminator.

See `docs/recipes/async-validation.md` for the full picture —
including async uniqueness checks, the `isValidating` UI pattern,
and how validation integrates with server-side errors.

## Additive: submission state, aggregates, reset, field arrays

None of these break existing code. They're new fields on the
`useForm` return value:

- `isDirty` / `isValid` — form-level aggregates (computed refs).
- `isSubmitting` / `submitCount` / `submitError` — submission lifecycle.
- `reset(next?)` / `resetField(path)` — restore form state.
- `append` / `prepend` / `insert` / `remove` / `swap` / `move` /
  `replace` — typed array helpers.

See `docs/api.md` for the full signature surface and
`docs/recipes/dynamic-field-arrays.md` for the array-helper usage
pattern.

## Additive: `sideEffects: false`

`@chemical-x/forms` now ships `"sideEffects": false` in its
`package.json`. Bundlers tree-shake unused exports — especially useful
if you only need one subpath in a large app.
